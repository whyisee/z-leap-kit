# whyisee.xyz 论坛系统完整功能设计

## 1. 文档目标

当前 `whyisee.xyz` 已经有了社区外观、分类、标签、话题详情、国际化和一个临时管理员登录入口。但这个登录只适合作为早期后台保护，不是一个真正论坛应该有的账号系统。

本文档用于定义一个完整论坛应该具备的产品能力、数据模型、权限体系、运营后台和开发路线，避免后续继续在空架子上堆功能。

设计目标：

- 让站点从“内容展示页”升级为“可发帖、可回复、可治理、可增长”的社区。
- 保持 2C2G VPS 可运行，不引入过重服务。
- 先支持站长可控运营，再逐步开放普通用户参与。
- 每个功能都要服务社区质量、SEO、用户留存或站长运营效率。

## 2. 产品定位

`whyisee.xyz` 是面向独立开发者、AI 工具用户、效率工具作者、SEO 实践者、小产品站长和内容站实践者的轻量社区。

一句话定位：

> 看见想法，验证想法，做出东西。

社区核心价值：

- 沉淀真实项目过程，而不是只发成果展示。
- 记录 AI 工具、独立开发、SEO、流量、开源插件、小产品变现的实践经验。
- 让用户能提问、复盘、求反馈、展示项目、跟进进展。
- 让旧内容能通过分类、标签、搜索、推荐和 SEO 持续产生价值。

## 3. 当前实现边界

当前已经具备：

- 首页、最新、分类、标签、项目展示、话题详情页面。
- 左中右三栏布局和基础深色论坛样式。
- 中文和英文切换。
- PostgreSQL 基础表：`users`、`categories`、`topics`、`posts`、`tags`、`topic_tags`、`reactions`、`bookmarks`、`audit_logs`。
- 临时管理员登录，依赖环境变量里的管理员账号密码。
- 管理后台入口和新建话题页面雏形。
- RSS、Sitemap、favicon、基础 SEO。

当前数据库配置：

```text
DB_HOST=127.0.0.1
DB_PORT=15432
DB_NAME=zi
DB_USER=postgres
DB_PASSWORD=123456
DB_SCHEMA=ws
```

当前缺失：

- 普通用户注册和登录。
- 密码哈希、会话表、找回密码、邮箱验证。
- 用户资料、头像、个人设置。
- 用户发帖、回复、编辑、删除。
- 通知、收藏、关注、点赞、举报。
- 审核队列和版主管理。
- 站内搜索。
- 后台内容管理、用户管理、举报处理和统计面板。
- 访问统计、社区活跃度统计、SEO 数据闭环。

结论：

> 当前登录只能算 V0.1 的管理员门禁。后续必须重构成完整账号体系，后台只是角色权限的一部分。

## 4. 核心用户角色

### 4.1 匿名访客

能力：

- 浏览公开话题。
- 查看分类、标签、项目展示、规则、关于页面。
- 使用公开搜索。
- 查看公开用户主页。
- 通过 SEO 页面进入站点。

限制：

- 不能回复。
- 不能发帖。
- 不能收藏、点赞、举报。
- 访问频率受限制。

### 4.2 新用户

能力：

- 登录后回复。
- 收藏话题。
- 点赞或感谢。
- 举报内容。
- 编辑自己的资料。

限制：

- 发帖需要邀请码、审核或满足条件。
- 每日回复和发帖数量较低。
- 上传附件受限。
- 链接数量受限。

### 4.3 普通成员

能力：

- 发布话题。
- 回复话题。
- 编辑自己的话题和回复。
- 收藏、点赞、关注话题。
- 上传图片或附件。

限制：

- 不能管理他人内容。
- 删除内容应是软删除或申请隐藏。
- 仍受反垃圾限制。

### 4.4 可信成员

能力：

- 更高发帖和回复额度。
- 可以帮助标记低质量内容。
- 可以参与精选推荐。
- 可以在部分分类免审核发布。

### 4.5 版主

能力：

- 隐藏、恢复、锁定、移动话题。
- 编辑分类内的话题标签。
- 处理举报。
- 管理违规回复。
- 查看部分审核日志。

### 4.6 管理员

能力：

- 管理用户、分类、标签、话题、回复、举报。
- 配置站点基础信息。
- 管理 SEO、Sitemap、RSS、邮件、备份。
- 查看访问统计和社区运营数据。
- 分配角色和权限。

## 6. 账号系统设计

### 6.1 登录方式

首版推荐：

- 邮箱或用户名加密码。
- 邀请码注册。
- GitHub OAuth 作为后续增强。

原因：

- 邮箱密码最容易自控。
- 邀请码适合早期控制社区质量。
- GitHub OAuth 对独立开发者友好，但要处理回调、绑定和账号合并。

### 6.2 注册流程

早期注册策略：

1. 用户进入 `/signup`。
2. 填写用户名、邮箱、密码、邀请码。
3. 校验用户名和邮箱唯一性。
4. 写入用户，状态为 `pending` 或 `active`。
5. 如果启用邮箱验证，发送验证邮件。
6. 验证通过后进入新用户状态。

可选策略：

- 邀请码有效时直接激活。
- 没有邀请码时进入申请队列。
- 管理员可以后台手动通过申请。

### 6.3 登录流程

1. 用户进入 `/login`。
2. 输入用户名或邮箱、密码。
3. 校验密码哈希。
4. 创建服务端 session。
5. 设置 HttpOnly Cookie。
6. 跳转到来源页或首页。

安全要求：

- 密码必须使用 `argon2id` 或 `bcrypt` 哈希。
- 登录失败需要限速。
- Cookie 设置 `HttpOnly`、`SameSite=Lax`，生产环境启用 `Secure`。
- 会话需要过期时间和主动注销能力。

### 6.4 找回密码

流程：

1. 用户提交邮箱。
2. 生成一次性 token。
3. 邮件发送重置链接。
4. 用户设置新密码。
5. 清理旧 token。
6. 使旧 session 失效。

### 6.5 用户资料

字段：

- 用户名。
- 显示名。
- 头像。
- 简介。
- 个人网站。
- GitHub 链接。
- 语言偏好。
- 通知偏好。
- 注册时间。
- 最后活跃时间。

页面：

- `/u/:username` 用户主页。
- `/settings/profile` 资料设置。
- `/settings/account` 账号安全。
- `/settings/notifications` 通知设置。

- `hidden`：已隐藏。

- 作者。
- 分类。
- 标签。
- 类型。
- 状态。
- 是否置顶。
- 是否精选。
- 发布时间。
- 最后活动时间。
- 浏览数。
- 回复数。

### 7.4 发帖编辑器

- 关注话题。

## 13. 审核和治理设计

### 13.1 举报

举报对象：

- 话题。
- 回复。
- 用户。

举报原因：

- 垃圾广告。
- 人身攻击。
- 违规内容。
- 重复内容。
- 错误分类。
- 其他。

处理状态：

- 待处理。
- 已接受。
- 已驳回。
- 已处理。

### 13.2 审核队列

进入审核队列的情况：

- 新用户首次发帖。
- 包含多个链接。
- 包含敏感词。
- 被多个用户举报。
- 短时间大量发帖。

审核操作：

- 通过。
- 驳回。
- 编辑后通过。
- 隐藏。
- 封禁用户。

### 13.3 反垃圾

策略：

- IP 限速。
- 登录失败限速。
- 发帖频率限制。
- 新用户链接数量限制。
- 敏感词规则。
- 邀请码。
- 管理员审核。

不建议首版接太复杂的反垃圾服务，先用规则和人工控制。

- `search_index` 或 PostgreSQL `tsvector` 搜索索引。

### 17.4 推荐会话表

```sql
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  user_id INTEGER NOT NULL,
  user_agent TEXT,
  ip_hash TEXT,
  expires_at TEXT NOT NULL,
  created_at TEXT NOT NULL,
  last_seen_at TEXT NOT NULL,
  revoked_at TEXT,
  FOREIGN KEY (user_id) REFERENCES users(id)
);
```

### 17.5 推荐通知表

```sql
CREATE TABLE notifications (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL,
  type TEXT NOT NULL,
  actor_id INTEGER,
  target_type TEXT NOT NULL,
  target_id INTEGER NOT NULL,
  metadata_json TEXT NOT NULL DEFAULT '{}',
  read_at TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id),
  FOREIGN KEY (actor_id) REFERENCES users(id)
);
```

### 17.6 推荐举报表

```sql
CREATE TABLE reports (
  id SERIAL PRIMARY KEY,
  reporter_id INTEGER NOT NULL,
  target_type TEXT NOT NULL,
  target_id INTEGER NOT NULL,
  reason TEXT NOT NULL,
  detail TEXT NOT NULL DEFAULT '',
  status TEXT NOT NULL DEFAULT 'pending',
  handled_by INTEGER,
  handled_at TEXT,
  created_at TEXT NOT NULL,
  FOREIGN KEY (reporter_id) REFERENCES users(id),
  FOREIGN KEY (handled_by) REFERENCES users(id)
);
```

## 22. 早期不做

为了避免一个人开发被复杂度拖死，早期不做：

- 私信。
- 实时聊天。
- 积分商城。
- 复杂等级自动晋升。
- 复杂富文本编辑器。
- 推荐算法。
- 支付系统。
- 多租户。
- 大规模分布式架构。

这些功能可以预留字段和接口思路，但不要进入前三个版本。

## 23. 设计结论

真正的论坛不是一个登录按钮，也不是几张列表页。它至少需要四条主线：

- 账号和权限：谁能做什么。
- 内容和互动：用户如何发帖、回复、收藏、关注。
- 治理和后台：站长如何控制质量和处理风险。
- 搜索和增长：内容如何被再次发现。

`whyisee.xyz` 的开发顺序应该是：

1. 先把内容社区跑顺。
2. 再做完整账号系统。
3. 再开放发帖回复。
4. 再做通知、审核、搜索和统计。

这样既不会一开始就被完整论坛复杂度拖死，也不会继续做只有界面的空架子。
